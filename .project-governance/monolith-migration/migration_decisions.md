### MD-001 — Module boundary naming: rename `livestock` → `poultry`?

- **Context:** the user's migration brief names the module "Poultry"; the existing service is `livestock-service` and BRD/SRS language and code use "livestock"/"batch" terminology throughout (tables, event schema names like `BatchOpenedEvent`).
- **Options:** (a) rename to `poultry` module during M2, (b) keep `livestock` internally and only use "Poultry" as the user-facing/doc label.
- **Decision:** RESOLVED in M2 — the module *directory/package* is named `app/poultry/` (matches the target architecture brief). Internal code is unchanged: class names (`Batch`, `BatchRepository`, etc.), table names (`batches`, `mortality_records`, ...), and event schema names (`BatchOpenedEvent`, ...) are left exactly as-is. Only the top-level package name changed, not the domain vocabulary inside it. This is a low-risk, purely structural rename.
- **Date:** 2026-06-18 (resolved 2026-06-18, M2 execution).

### MD-002 — RabbitMQ / event infrastructure: keep or drop?

- **Context:** audit (`current_state_architecture_report.md` §3) found RabbitMQ pub/sub is 100% unused at runtime today — no service publishes or consumes a single event despite full scaffolding existing in every service.
- **Options:** (a) drop RabbitMQ entirely in M3/M7, replace with nothing (modules call each other in-process when needed); (b) implement a minimal in-process event bus in M3 using the existing `shared/events/schemas.py` definitions, primarily so `notifications` can finally become a real consumer.
- **Decision:** RESOLVED in M3 → **(a), drop.** Reversed the earlier lean toward (b): building a working event bus where none currently exists is a *new feature* (making notifications event-driven for the first time), not a migration of existing behavior — out of scope for an architecture migration whose mandate is to preserve functionality, not add to it. If event-driven notifications are wanted later, that is a product decision for a separate initiative, made with full BRD/SRS traceability, not a side effect of this migration.
- **Action taken (M3):** the 8 `publisher.py` files copied into `app/*/infrastructure/messaging/` were marked DEPRECATED with a banner explaining why (verified zero call sites repo-wide) and citing the M7 deletion gate. They are not yet deleted — anti-destruction rule — and the original `services/*/infrastructure/messaging/publisher.py` files were left completely untouched. The RabbitMQ container in `docker-compose.yml` is untouched; its removal is M7 scope, gated on M6 (runtime simplification) confirming nothing depends on it.
- **Date:** 2026-06-18 (resolved 2026-06-18, M3 execution).

### MD-003 — `farms_ref` duplication (identity-service, livestock-service)

- **Context:** both services have a denormalized `farms_ref` table intended to be populated by a `FarmCreatedEvent` consumer that does not exist; the data is therefore unpopulated/stale today, independent of this migration. M2 confirmed (`app/identity/domain/models/user.py`, `app/poultry/domain/models/animal.py`) that `farms_ref` is not just an inert table — `users.farm_id` and `batches.farm_id` carry a live `ForeignKey("farms_ref.id")` constraint, so it cannot simply be dropped without replacing the FK target.
- **Options considered:** (a) once identity, farm, and poultry modules share one database (M4), repoint `users.farm_id` and `batches.farm_id` FKs from the local `farms_ref.id` to the real `farm` schema's `farms.id` (cross-schema FK, legal within one Postgres database) and drop the two `farms_ref` tables; (b) keep `farms_ref` as a read-only projection table, synced via the event bus from MD-002 — moot now that MD-002 resolved to "drop the event bus."
- **Decision:** RESOLVED → (a). Once M4's single database with schema-per-module exists, repoint the FKs at `farm.farms.id` and drop both `farms_ref` tables.
- **Execution (update):** Docker/DB access was restored mid-session (the user supplied sudo credentials after the initial blocker was reported). Executed against the live `agrovision` database: confirmed zero orphaned `farm_id` values in `identity.users`/`poultry.batches` against `farm.farms` (i.e., every reference already resolves to a real farm — the FK repoint introduces no broken references), then in one transaction: dropped `users_farm_id_fkey`/`batches_farm_id_fkey`, re-added them pointing at `farm.farms(id)`, dropped `identity.farms_ref` and `poultry.farms_ref`. Verified post-change via `pg_constraint` and a join spot-check. The original `identity_db`/`livestock_db` databases (and their `farms_ref` tables) were left completely untouched — this only affected the new consolidated schemas.
- **Execution (M5 follow-up):** loading both `app.identity` and `app.poultry` into the same Python process (M5) surfaced that both modules still defined a `FarmRef` ORM class mapped to the now-dropped `farms_ref` table, which collided in `shared.models.base.Base`'s shared `MetaData` registry (`sqlalchemy.exc.InvalidRequestError: Table 'farms_ref' is already defined`). Confirmed via grep that `FarmRef` was referenced nowhere outside its own class definition in either module, then removed the class from both `app/identity/domain/models/user.py` and `app/poultry/domain/models/animal.py`, and changed `users.farm_id`/`batches.farm_id`'s `ForeignKey(...)` string from `"farms_ref.id"` to `"farm.farms.id"` — code finally matching the DB schema that M4 already established. Verified by importing `app.main` successfully and by live API calls returning correct farm-linked data.
- **Date:** 2026-06-18 (resolved and executed; ORM-level follow-up also 2026-06-18, during M5).

### MD-004 — Alembic migration strategy across modules

- **Context:** each service currently has its own independent Alembic history (identity: 2 revisions, farm: 1, livestock: 3, inventory: 1, finance: 2, notification: 1, reporting: 0), each with its own `alembic_version` table inside its own database.
- **Options considered:** (a) one consolidated Alembic project with one linear revision history across all modules; (b) one Postgres database, one schema per module, each module keeping its **existing, unmodified migration scripts**, with Alembic's `version_table_schema` pointed at that module's schema so each module's `alembic_version` table lives inside its own schema (`identity.alembic_version`, `farm.alembic_version`, etc.) and migrations run with the connection's `search_path` set to that schema.
- **Decision:** RESOLVED → (b). This preserves every existing migration file completely unchanged (lowest risk, easiest to verify, easiest to reverse) and only changes *where* each module's tables land (a schema inside one DB, instead of a whole separate DB) and *how* its Alembic state is tracked (per-schema version table instead of per-database). A single linear history (option a) would require rewriting migration dependency chains across 7 previously-independent histories for no functional benefit — rejected as needless risk for this migration's goals.
- **Date:** 2026-06-18.

### MD-006 — Location and import-compatibility of `shared/` in the monolith

- **Context:** `target_architecture.md` §2 left open whether `shared/` moves into `app/shared/` or stays at repo root.
- **Decision:** RESOLVED in M2 — `shared/` **stays at the repo root, unmoved**. Every copied module's `shared.*` imports (`shared.contracts.api_standards`, `shared.exceptions`, `shared.models.base`, `shared.events.schemas`, `shared.utils.*`) were left untouched during the M2 import-rewrite (verified: zero `shared.` import lines were modified). This works because `app/` and `shared/` are sibling top-level packages under the repo root, exactly as `services/<name>/app` and root `shared/` are siblings today (same `PYTHONPATH` convention). No code change required to `shared/` itself.
- **Date:** 2026-06-18.

### MD-007 — Internal auth trust boundary inside the monolith (discovered in M5)

- **Context:** while smoke-testing M5, the `/api/v1/reports/batch/{id}` endpoint started returning 500 (caused by a 401 from the in-process call to `/api/v1/batches/{id}`). Root cause: the original architecture had the JWT/header-injection check happen *only* at the api-gateway; the direct reporting→livestock/finance httpx calls (audit §2) bypassed the gateway entirely and were never re-authenticated downstream (services trust headers, don't re-verify). My first M5 implementation routed the in-process client calls through the *same* single ASGI app object that also carried the new app-wide JWT middleware, which incorrectly applied that protection to what used to be an unauthenticated internal hop — a behavior change, not a faithful migration.
- **Decision:** split `app/main.py` into two ASGI objects: `fastapi_app` (routers + CORS + exception handlers, no JWT middleware — the equivalent of "the 7 microservices, which never re-verified JWTs") and `app` (`fastapi_app` wrapped in `AuthHeaderInjectionMiddleware` — the equivalent of "the gateway sitting in front"). External traffic is served via `app`; reporting's in-process clients call `fastapi_app` directly, exactly reproducing the old trust boundary (external requests are authenticated once at the edge; internal service-to-service calls are not re-authenticated).
- **Verification:** live-tested both the failure (before the fix: 500/401) and the fix (after: full report endpoint returns 200 with correct aggregated data, byte-identical to the real running `reporting-service`'s output for the same batch).
- **Date:** 2026-06-18.

### MD-008 — Alembic-on-startup for the monolith: defer

- **Context:** every old service runs `alembic upgrade head` against its own DB in its `main.py` lifespan (`_run_migrations()`, pointed at `/app/alembic.ini`). M2 did not copy each service's `migrations/`/`alembic.ini` into `app/<module>` (only the `app/` subtree was copied — see migration_backlog.md MIG-M2 scope). M4 already brought every consolidated schema to the correct state directly (`pg_dump`/restore carried over each module's `alembic_version` table as part of the data copy), so there is no pending migration to apply.
- **Options considered:** (a) build a proper consolidated Alembic setup now (copy each module's `migrations/` into `app/<module>/migrations`, write schema-aware `env.py` per module with `version_table_schema`+`search_path`, wire `alembic upgrade head` per module into the monolith's lifespan); (b) defer — the monolith boots without running any migrations, since M4 already left every schema at the correct version.
- **Decision:** RESOLVED → (b), defer to a later, dedicated pass. Building 6 schema-aware Alembic environments and testing them properly is real, separate engineering work, not required to prove M6's actual goal (the monolith can run as a single container against the consolidated DB). Doing it hastily here risks shipping an untested migration runner. Tracked as a follow-up item, not a blocker for M6.
- **Date:** 2026-06-18.

### MD-009 — When to mark old service containers deprecated

- **Context:** M6 proved the monolith container runs and reproduces the live system's behavior for a manually-tested subset of endpoints (auth, farms, batches, the cross-module report in JSON+PDF, plus a few error paths). The original 8 service containers are still the only ones serving real traffic (via nginx/gateway).
- **Options considered:** (a) mark the 8 old containers "deprecated" in docker-compose now, on the strength of M6's manual testing; (b) wait until M8 (running the full existing automated E2E suite — `tests/e2e/*`, currently 15 tests across livestock/finance/reporting workflows — plus the UAT script from P-15, against the monolith) before making any deprecation decision.
- **Decision:** RESOLVED → (b). A manual spot-check of ~10 requests is not sufficient evidence to start deprecating a live, working system — that's exactly the kind of premature action the migration's anti-destruction rule and "verify, don't trust" rule exist to prevent. Deprecation marking will be reconsidered once M8 has run the full automated suite against the monolith and the results are recorded.
- **Date:** 2026-06-18.

### MD-010 — Why `tests/e2e/*` was not force-fitted to run against `app/<module>` in M8

- **Context:** M8 needs E2E coverage of the monolith. `tests/e2e/{finance,livestock,reporting}/conftest.py` makes this hard: each fixture deletes `app`/`shared` from `sys.modules`, removes any `services/` path entry, then inserts exactly one `services/<name>-service` path, and the test files themselves import unprefixed (`from app.domain.models.animal import ...`). The monolith's modules are namespaced (`app.poultry.domain...`), so this import never resolves against `app/`.
- **Options considered:** (a) hack around it with a `sys.modules` aliasing trick (e.g. `sys.modules["app"] = app.poultry` before each test) so the unprefixed imports resolve to the namespaced module; (b) leave the existing tests untouched (they keep validating `services/*`, which is still the live system) and treat monolith E2E coverage as separate, real follow-up work (rewriting the tests to be either parametrized or HTTP-based).
- **Decision:** RESOLVED → (b). Option (a) is fragile: aliasing the top-level `app` module in `sys.modules` to `app.poultry` would break any other code in the same test process that expects the real top-level `app` package (e.g. `app.core.config`), and produces a shim that "passes" without actually proving the monolith's own code paths behave correctly under the test's specific mocking setup — false confidence is worse than an honestly-reported gap. Rewriting the tests properly (CLEAN-05 in `repository_cleanup_backlog.md`) is real engineering work and should be scoped and done deliberately, not improvised under an M8 time-box.
- **What WAS done instead in M8:** ran the existing `tests/e2e/*` suite unmodified (15/15 pass — confirms the migration hasn't broken the still-live `services/*` side) and supplemented it with **direct, manual API-level verification of every UAT test case (TC-01 through TC-09) against the live running monolith container** — a more thorough check than the existing E2E suite would have given even if it were portable, since it exercises the actual HTTP layer, the consolidated database, and real business-rule responses (e.g. confirmed the quarantine-period rule, the active-batch-only rules for feed/mortality/weight, and the profit calculation arithmetic, all matching expected behavior).
- **Real bug found and fixed via this manual testing (see MD-011):** batch creation crashed with `NoReferencedTableError` — something none of M5/M6's testing had caught because no prior test exercised a `POST /batches/`.
- **Date:** 2026-06-18.

### MD-011 — Cross-schema FK resolution bug found in M8, fixed

- **Context:** `app/farm/domain/models/farm.py`'s `Farm`/`Building`/`Section` models never declared `schema="farm"` in `__table_args__`. M5's FK repoint (MD-003) made `identity.users.farm_id` and `poultry.batches.farm_id` reference the literal string `"farm.farms.id"`, but since `Farm`'s SQLAlchemy `Table` object was registered under metadata key `"farms"` (no schema prefix), `"farm.farms"` never matched it.
- **Why M5/M6 testing missed this:** all prior verification was read-only (`GET` requests) — Postgres resolves unqualified-vs-qualified table names via `search_path` at the SQL level regardless of SQLAlchemy's Python-side metadata, so `SELECT`s worked fine. The bug only manifests when SQLAlchemy needs to topologically sort tables for an `INSERT`'s FK dependency ordering — i.e., only on writes to a table with a cross-schema FK. `POST /api/v1/batches/` is exactly that, and nothing before M8 had tested it against the monolith.
- **Fix:** added `__table_args__ = {"schema": "farm"}` to `Farm`, `Building`, and `Section`, and fully schema-qualified their FK strings (`"farm.farms.id"`, `"farm.buildings.id"`) so SQLAlchemy's metadata-level table identity matches at the Python level, not just the Postgres `search_path` level.
- **Verification:** rebuilt and restarted `agrovision-monolith-1`; `POST /api/v1/batches/` (with a real farm/section) now succeeds (201) where it previously raised `NoReferencedTableError`. Re-ran the rest of the UAT pass (feed, mortality, weight, expense, sale, profit — all on a real existing active batch) successfully afterward.
- **Lesson for future phases:** read-only verification is not sufficient evidence that cross-schema FK wiring is correct — any future schema work must include at least one write test per cross-schema relationship.
- **Date:** 2026-06-18.

### MD-012 — M7 execution: explicit user override of the MD-009 deferral, and pre-execution gap-closing

- **Context:** MD-009 deferred any deprecation/cutover decision until M8's evidence was in hand. M8 finished MOSTLY complete (real bug found and fixed, MD-011) but with two explicit known gaps: `app/inventory` and `app/notifications` write paths were never exercised by a live request (only farm/identity/poultry/finance got real write testing), and TC-10/literal-E2E-portability were recorded gaps (MD-010). The user then explicitly instructed "Complete M7 also," overriding the MD-009 deferral.
- **Decision:** treat the explicit instruction as authorization to proceed, but first close the one gap that was cheaply closeable before any irreversible step: write-test `app/notifications` (inventory's write paths — receive/dispatch — had already been exercised in the M8 UAT pass per `migration_verification.md` TC-06, so only notifications was a genuine gap).
- **What was found:** `app/notifications`'s create/list endpoints return 500 (`UndefinedColumnError: notifications.created_by does not exist`) — the ORM model uses `AuditMixin` (adds `created_by`/`updated_by`) but the actual `notifications` table (in both the original `notification_db` and the migrated `notifications` schema) never had those columns. Reproduced the identical 500 against the still-live, original `notification-service` via the real gateway (`http://localhost`) before touching anything — confirming this is a **pre-existing bug in the original microservice**, not a migration regression. Fixing unrelated pre-existing defects is out of scope for an architecture migration (same reasoning as MD-002); left as-is, documented here and in `migration_status.md` for whoever picks it up next.
- **Then executed M7 in full**, each step backed by a verification check immediately after (see `migration_status.md` verification log and `migration_verification.md` Deletion Log): backed up all 7 original + the consolidated database first; cutover (nginx repoint); stopped the 8 old containers (with a full smoke test before progressing to the next, irreversible step); dropped the 7 old databases; removed RabbitMQ; deleted `services/*`; fixed the resulting breakage in `scripts/seed/config.py` (a real, direct consequence of the DB drop that would otherwise have left the repo in a broken state); updated all developer-facing docs (`README.md`, `DEPLOYMENT.md`, `Makefile`, `docker-compose.dev.yml`, `.env.example`) to stop describing an architecture that no longer exists.
- **Date:** 2026-06-18.
